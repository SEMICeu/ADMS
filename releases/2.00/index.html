Introduction

This document describes a vocabulary, in this case ADMS Vocabulary. A vocabulary is a semantic data specification designed so that the included terms are highly reusable. The included classes and properties come with a limited set of constraints.

ADMS, the Asset Description Metadata Schema, is a vocabulary for describing semantic assets (or just 'Assets'), defined as highly reusable metadata (e.g. xml schemata, generic data models) and reference data (e.g. code lists, taxonomies, dictionaries, vocabularies) that are used for eGovernment system development.

Motivation for this vocabulary

Someone searching for a Semantic Asset is likely to have different needs, priorities and expectations than someone searching for a dataset in a data catalog and these differences are reflected in ADMS. In particular, users seeking an Asset are likely to be searching for a document — something they can open and read using familiar desktop software, as opposed to something that needs to be processed. Of course this is a very broad generalization. If a code list is published as a SKOS Concept scheme [[skos-reference]] then it is both an Asset and a dataset offered to be exploited in a data exchange. As vocabulary ADMS focusses on the nature of the Assets, rather than on the organisation of the Assets in a catalogue.

Although ADMS has been designed from the above usage case in mind, some terms are used beyond that use case and operate in a broader context. For instance adms:identifier got a wide adoption in various vocabularies such as DCAT-AP [[dcat-ap]] and the SEMIC Core Vocabularies. In this new release of ADMS this usage has been made more visible, by decoupling ADMS in a formal sense from DCAT and restricting the specification to the classes and properties defined in the ADMS namespace. The historic connection with DCAT has been reformulated an as usage example of ADMS.

The namespace for ADMS is http://www.w3.org/ns/adms# .

The vocabulary is also provided in different RDF serializations [ turtle | ntriples | rdf/xml ].

History

ADMS was first developed by and published by the European Commission ISA Programme. Contributors included representatives of Member States of the European Union, operators of national repositories, standardisation bodies and independent experts whose work was published in April 2012 [[ADMS1]]. That document includes the original history and motivation behind the development of ADMS, as well as the business need and usage scenario for it.

Further development and review has been undertaken by the Government Linked Data Working Group [[WG-GLD]]. This version of ADMS builds on the original work in a broader, global context. It also includes some changes to the original version made as a result of implementation experience, particularly on the European Commission's Joinup Platform. Under supervision of this Working Group a minor update has been published, which did not include any substantial changes compared to the first published version by the Working Group. The latest document was published as a W3C Working Group Note [[vocab-adms]] at 1 August 2013 and has not been revised since than.

Motivation for this version

In the light of providing the semantic data community, and thus especially the SEMIC community, a coherent and trusted collection of maintained vocabularies, the European Commission ( Interoperable Europe ) engaged itself to take up the active maintenance of the ADMS vocabulary. In the past decade the SEMIC community of the European Commission has been supporting ADMS by taking up the management of the associated controlled vocabularies and the management of an application profile, independently from the W3C note. Given that initiative for this vocabulary as been taken by the European Commission and this vocabulary has not been adopted as the W3C recommendation, W3C and the European Commission collaborate together to provide an active maintenance of this vocabulary.

This new release of ADMS will decouple ADMS from DCAT for the terms in the vocabulary. Doing so the lifecycle of ADMS is not anymore bound to the lifecyle of DCAT. With the active development of DCAT and its profiles, the status of ADMS as profile of DCAT becomes unclear. The impact of DCAT on the ADMS vocabulary as profile of DCAT is not only on its relationship, but also on how the profile is constructed and on the adoption of some terminology. For instance, ADMS has some versioning navigation properties, but those are not adopted by DCAT but replaced by similar properties in the DCAT namespace. All these changes to the base vocabulary DCAT, of which ADMS declared itself originally a profile of, force a regular adaption of ADMS.

With this new release ADMS is decoupled to the furtest extend possible from DCAT (no formal ties), while maintaining its original definitions. In essence one can consider this as a broadening of the ADMS terms.

While this independence offers new potential for reuse, the goal is not to create a parallell universe to DCAT. As such it is highly recommended in order to use ADMS in full, to first create a DCAT profile with the ADMS terms included. This means that although some ADMS classes and properties are not deprecated, the newly created DCAT profile will adhere first to DCAT, and only use the ADMS terms that are fit for purpose for the profile. The historic depencency should facilitate the creation of such profile easily.

ADMS is a vocabulary that defines a collection of terms having minimal usage restrictions. Usage of the terms is considered conformant with ADMS if the terms (classes and properties) from ADMS are utilised in a manner that aligns with their semantics as specified in ADMS. That means the usage is conformant to

  • the definitions and usage notes for the classes and properties, and
  • the domain and ranges of the properties
In this vocabulary the rdfs:Resource class is used to denote any possible resource. Thus meaning no restriction in usage.

Historic interpretation of ADMS in the context of DCAT.

ADMS was originally very tightly integrated with DCAT. As motivated in the introduction [[[#specintro]]], this version decouples ADMS from DCAT. Nevertheless, this historic connection is important. This section, therefore, provides the base principles for building an ADMS based profile of DCAT.

The principles are:

  • The main classes adms:AssetRepository, adms:Asset and adms:AssetDistribution are respectively defined as subclasses of dcat:Catalog, dcat:Dataset and dcat:Distribution.
  • Use the relations dcat:dataset (or the more generic dcat:resource) and dcat:distribution to interconnect adms:AssetRepository, adms:Asset and adms:AssetDistibution.
  • The properties adms:last, adms:next, adms:prev have dcat:last, dcat:next, dcat:prev counterparts. It is recommended to use the DCAT ones in favor of the adms counterparts.
The last principle formulates that if the objective is to create a profile of DCAT, it recommended to the best extend with the DCAT specification. ADMS terminology should only be adopted in the profile when they precisely capture the needs.

Example

This example illustrates the use of ADMS in the orginal context of a DCAT profile. Examples in this document are serialized in Turtle [[turtle]].

First, the repository description: 


:Repository a adms:AssetRepository ;
  dcterms:created "1947-02-07"^^xsd:date ;
  dcterms:description "A complete catalog of code lists and standards created by the Exemplary Standards Body"@en ;
  dcterms:publisher  ;
  dcterms:title "The Exemplary Standards Body Catalog"@en .

Example 1 - ADMS as DCAT profile

This assumes that the Exemplary Standards Body is described at http://example.com/data#org . Next, an Asset. We'll create an imaginary code list called 'Fruit I like' for our example:
  • apples
  • pears
  • oranges
  • peaches
We might describe this Asset thus: 
  :Fruit_02 a adms:Asset ;
    dcterms:created "1999-05-24" ;
    dcterms:description "Fruits that are found to be generally liked by most people." ;
    dcterms:publisher  ;
    dcterms:title "Fruit I like"@en ;
    adms:status  ;
    dcterms:type  ;
    adms:previous :Fruit_01 ;
    adms:last :Fruit ;
    dcat:distribution :Fruit_02.csv ;
    dcat:distribution :Fruit_02.xml .
This provides the creation date (line 2), description (line 3), publisher (line 4) and a title (line 5) for the Asset. The status of the Asset is given in line 6 using one of the values made available in the codelists created in the context of the first version of ADMS [[codelist-adms]] . Likewise the type of Asset in line 7. In line 8 we can see that there was a previous version of this asset (Fruit_01) and that the latest version can be found by appending 'Fruit' to the data's root (line 9). This Asset has two distributions linked in lines 11 and 12. One of these might be described thus: 
:Fruit_02.xml a adms:AssetDistribution ;
  dcterms:description "XML encoding of Fruits that are found to be generally liked by most people." ;
  dcterms:license  ;
  dcterms:format  .
This is a very simple description of the Asset Distribution that just gives its description, format, and license. In this example we've used the NAL File Type [[NAL-FileType]] to provide a URI for the XML MIME type. The missing triple from this example is the one that includes the Asset in the Repository. This is done with a simple triple: 
:Repository dcat:dataset :Fruit_02 .
